MARA

Multi-Agent Risk-Adjusted Capital Orchestrator

An open-source autonomous trading system that coordinates specialized AI agents across crypto, futures, and prediction markets. Capital flows dynamically based on real-time macro regime classification, conflict indexing, and per-worker risk scoring. The whole stack runs on a Raspberry Pi 5.

What It Does

MARA is built around one idea: different market conditions call for different strategies. A system that trades crypto momentum during a bull run should not be running the same playbook when geopolitical conflict spikes or a recession sets in.

A central FastAPI Hypervisor runs a 60-second loop: pull market data, classify the current macro regime, allocate capital to whichever workers are best suited for that regime, enforce risk limits, and execute. Five specialized workers handle the actual trading, each designed for a specific edge. The hypervisor never trades directly. It orchestrates.

The system runs entirely on Docker containers. Dev happens on WSL2 Ubuntu, production deploys to a Raspberry Pi 5 via ARM64 images built with docker buildx.

Every worker implements the same REST contract: /health, /status, /metrics, /signal, /execute, /allocate, /pause, /resume, /regime. The hypervisor doesn't care what's behind the API. Swap NautilusTrader for a different engine and nothing else changes.

Workers

Nautilus

MACD + Williams Fractals swing trading on OKX perpetuals. Dual-confirmation entry with RSI gating, trailing stop to breakeven at 1:1 R:R. Falls back to an internal paper simulator if the NautilusTrader wheel isn't available for the target platform.

Polymarket

CLOB market-making on prediction markets. Regime-gated exposure: expands during WAR_PREMIUM (high uncertainty = good spreads), pauses entirely in CRISIS_ACUTE (liquidity dries up). Wraps an existing open-source market maker bot.

AutoHedge

LLM-based advisory pipeline. Director, Quant, and Risk agents run sequentially on phi3:mini via Ollama. Advisory only, never executes. Each regime gets a tailored system prompt so the agents reason in context. Falls back to direct Ollama calls if the agent framework fails.

Arbitrader

Cross-exchange price arbitrage. Delta-neutral: long on the cheaper exchange, short on the more expensive. Python sidecar manages the JVM lifecycle for the Java arbitrage engine. In paper mode, the JVM is never launched; the sidecar simulates arb PnL directly.

StockSharp (Phase 2)

Interactive Brokers order routing via .NET 8 wrapper. Not yet implemented. Will handle US equities, options, and futures through IBKR's TWS API once the paper trading phase validates the rest of the stack.

Regime Classifier

The regime classifier decides how the system behaves. It fuses market data (VIX, DXY, gold/oil ratio, BDI, defense ETF momentum) with geopolitical conflict scoring from ACLED and GDELT to produce one of seven regime states. Market proxy data carries 70-75% of the weight. Conflict indices fill in the gaps that pure market data misses.

Key design choice: requiring 2-of-N signal confirmation before triggering a regime change. A single indicator spiking (say gold jumps on a supply disruption) won't flip the system into WAR_PREMIUM by itself. This prevents false positives from commodity-specific events.

Risk Manager

Seven enforcement points run every cycle. If any trip, the system either halts the offending worker, blocks new signals, or puts the entire portfolio into cooldown.

Portfolio-Level

  • Drawdown gate: Portfolio DD > 20% triggers 1-hour cooldown
  • PnL floor: Realized PnL below -$40 triggers cooldown
  • Max positions: More than 6 open positions blocks new signals
  • Free capital: Less than 15% cash reserve blocks new signals

Per-Worker

  • Allocation cap: No single worker gets more than 50% of capital
  • Worker drawdown: Worker DD > 30% excludes it from allocation
  • Sharpe penalty: Worker Sharpe below 0.5 cuts allocation by half

Data Sources

Market Data

  • OHLCV: yfinance (equities, ETFs, crypto)
  • Macro: FRED (fed rates, 10Y-2Y yield, DXY)
  • Commodities: Gold, oil, BDI proxy via BDRY ETF
  • Sentiment: VIX, BTC funding rates

Conflict Index

  • ACLED CAST: 21-day conflict forecast (weighted 20%)
  • ACLED Live: Recent conflict events by country (5%)
  • GDELT: Global news sentiment, Goldstein scale (5-25%)
  • GDELT is confirmation-only scoring, rate-limited

Key Design Decisions

  • OKX only: Binance returns HTTP 451, Bybit returns 403. Both geo-blocked. OKX works. Symbol format: BTC-USDT-SWAP
  • BDRY as BDI proxy: The Baltic Dry Index (^BDI) was delisted from yfinance. BDRY ETF tracks it well enough for regime classification
  • Market proxy primary (70-75%): Prevents commodity-only spikes from false-triggering WAR_PREMIUM. Conflict data is confirmation, not primary signal
  • 2-of-N confirmation: No single indicator can flip the regime by itself. Prevents false positives from isolated market events
  • AutoHedge advisory-only: Stripped the Execution agent (Solana-only, unusable). Director + Quant + Risk agents remain for LLM-based portfolio commentary
  • Workers as FastAPI services: Pluggable architecture. The hypervisor talks REST and doesn't care what runs behind the API. Swap engines without touching orchestration
  • Paper capital frozen: In paper mode, total capital stays at INITIAL_CAPITAL_USD. PnL is tracked but never compounds into the capital base. Prevents unrealistic snowball effects during testing
